## Vision

### 10x Check
Route ALL AI through the Next.js server (kill client-side key exposure). Build a Phase 1 validation gate so you know precisely when to flip the web switch. Ship the confidence routing UI so dead-zone captures become training data instead of silent noise. Add Supabase Realtime so the web dashboard feels alive. Set preferred nav app once, get friction-free navigation everywhere.

The version that makes Capsave a platform: one AI contract, both platforms, validated by real DAU data, with every user interaction feeding the pipeline back.

### Core Product Thesis
Screenshot = intent signal, not image file. The whole game is: capture intent on mobile, take action from anywhere. Remove every friction between that intent and the action.

### CEO Expansion 1: Mobile AI Unification

**API contract:**
```
POST /api/analyze
Content-Type: multipart/form-data
Authorization: Bearer <supabase_jwt>
MAX_IMAGE_SIZE_BYTES = 1_048_576  (1MB — server rejects above this with 413)

Body: { image: File }

Response 200: { type, name, places, confidence, source_account_id, ... }
Response 401: { error: "Unauthorized" }
Response 413: { error: "Image exceeds 1MB limit" }
Response 500: { error: "Analysis failed" }
```

**Auth:** Mobile sends Supabase session JWT in `Authorization` header. Server verifies via `createServerClient` before processing. Rejects unauthenticated requests with 401.

**Latency mitigation:** Adding server hop introduces ~500ms–2s latency vs. direct client call. Mitigations: (a) show indeterminate progress bar immediately on upload; (b) compress image to <1MB on mobile before upload; (c) set 30s server-side timeout with "Analysis is taking a while — try again" on timeout.

### CEO Expansion 2: Phase 1 Validation Gate

**Prerequisites (Supabase Dashboard → Database → Extensions):**
- Enable `pg_cron` extension
- Enable `pg_net` extension
- Set Resend API key: `ALTER DATABASE postgres SET app.resend_api_key = 'your_key_here'`
  (or use Supabase Vault: `SELECT vault.create_secret('resend_api_key', 'your_key')`)
- **REQUIRED before enabling cron:** `INSERT INTO app_config (key, value) VALUES ('owner_email', 'your@email.com')` — missing seed means alert fires silently with no email sent

**Mechanism:** pg_cron runs hourly. Uses `pg_net` HTTP extension to call Resend directly (two-step chain, no Edge Function deployment needed).

```sql
-- analytics.dau table (part of Migration 003)
CREATE TABLE analytics.dau (
  date DATE PRIMARY KEY DEFAULT CURRENT_DATE,
  distinct_users INTEGER NOT NULL DEFAULT 0,
  notified_at TIMESTAMPTZ DEFAULT NULL
  -- updated_at intentionally omitted — not queried anywhere in Phase 2
);

-- app_config table (part of Migration 003)
CREATE TABLE app_config (
  key TEXT PRIMARY KEY,
  value TEXT NOT NULL
);

-- pg_cron job (runs hourly)
-- Atomic CTE: upsert DAU count + send alert + mark notified in one transaction
SELECT cron.schedule('dau-counter', '0 * * * *', $$
  WITH upserted AS (
    INSERT INTO analytics.dau (date, distinct_users)
    SELECT CURRENT_DATE, COUNT(DISTINCT user_id)
    FROM captures
    WHERE created_at::date = CURRENT_DATE  -- calendar day UTC, NOT rolling 24h
    ON CONFLICT (date) DO UPDATE
      SET distinct_users = EXCLUDED.distinct_users
    RETURNING date, distinct_users, notified_at
  ),
  threshold AS (
    SELECT u.date, c.value AS owner_email
    FROM upserted u, app_config c
    WHERE u.distinct_users >= 10
      AND u.notified_at IS NULL
      AND c.key = 'owner_email'
  ),
  notified AS (
    UPDATE analytics.dau SET notified_at = now()
    WHERE date IN (SELECT date FROM threshold)
    RETURNING date
  )
  SELECT net.http_post(
    url     := 'https://api.resend.com/emails',
    headers := jsonb_build_object(
      'Authorization', 'Bearer ' || current_setting('app.resend_api_key'),
      'Content-Type', 'application/json'
    ),
    body    := json_build_object(
      'from', 'capsave@yourdomain.com',
      'to', owner_email,
      'subject', 'Phase 1 achieved!',
      'text', 'Capsave hit 10 DAU. Time to flip the web switch.'
    )::text
  )
  FROM threshold
  WHERE EXISTS (SELECT 1 FROM notified);
$$);
```

**Idempotency:** `notified_at` is set in the same CTE as the email send — atomic, no duplicate emails on retry.

**DAU definition:** Distinct `user_id` values in `captures` inserted on the current UTC calendar day (`created_at::date = CURRENT_DATE`). Not a rolling 24-hour window.

**Retention:** No retention policy required at current scale. Revisit at 90 days.

**Cron errors:** pg_net failures are logged to `net._http_response`. Monitor for non-2xx responses if needed.

### CEO Expansion 3: Uncertain Queue UI

**Trigger:** `confidence < 0.5` on a saved capture.

**UI behavior:**
- Captures with `confidence < 0.5` are grouped in a collapsible "Review needed" section at the TOP of the capture list (not inline badges on individual cards — cleaner when many are uncertain)
- Section header: "Review needed (N)" — tap to expand/collapse
- Each uncertain capture in the section shows a reclassify drawer on tap
- Drawer: "We're not sure about this one. What is it?" + 3 options: Place / Text / Dismiss
- After reclassification: capture moves out of "Review needed" into the normal list
- Tap outside drawer = dismiss without action (no reclassification)

**On reclassify (Place):**
1. Call `POST /api/geocode` synchronously with existing `name` field
2. On geocode success: `type = 'place'`, update `places` with geocode result, `confidence = 1.0`, `reclassified_at = now()`
3. On geocode failure: show toast "Geocoding failed — saved as place anyway"; still update `type`, `confidence`, `reclassified_at` (user's explicit intent overrides AI uncertainty)

**On reclassify (Text):** `type = 'text'`, clear `places` column, `confidence = 1.0`, `reclassified_at = now()`

**On Dismiss:** soft-delete: set `deleted_at = now()` (not a hard delete)

**Columns written:** `type`, `places`, `confidence`, `reclassified_at`, `deleted_at`

### CEO Expansion 4: Preferred Nav App Setting

**Storage:**
```sql
CREATE TABLE user_preferences (
  user_id UUID PRIMARY KEY REFERENCES auth.users(id),
  preferred_nav_app TEXT DEFAULT 'tmap'
    CHECK (preferred_nav_app IN ('tmap', 'naver', 'google', 'kakao')),
  updated_at TIMESTAMPTZ DEFAULT now()
);
```

**Upsert call (client):**
```typescript
await supabase.from('user_preferences')
  .upsert({ user_id, preferred_nav_app: selected }, { onConflict: 'user_id' })
```

**Context:** Preferences loaded into a `UserPreferencesContext` provider (new file: `apps/web/src/contexts/UserPreferencesContext.tsx`). Wraps the root layout alongside `CapturesContext`. `PlacePopup` reads `useUserPreferences().preferredNavApp`.

**Navigation hierarchy:**
- Web: `/settings/navigation` — accessible via sidebar settings icon
- Mobile: stack screen `SettingsNavigationScreen` — accessible via Profile tab → Settings

**Rendering rule:** Preferred app = filled primary button (full width). Other 3 = secondary outline buttons in a compact row below. If no preference set, default = Tmap.

### CEO Expansion 5: Supabase Realtime Sync

```typescript
// CapturesContext (web) — inside useEffect
const channel = supabase
  .channel('captures-changes')
  .on('postgres_changes',
    { event: 'INSERT', schema: 'public', table: 'captures',
      filter: `user_id=eq.${user.id}` },
    (payload) => setCaptures(prev => [payload.new as Capture, ...prev])
  )
  .subscribe()

// Cleanup — must be in useEffect return to prevent connection leak
return () => { supabase.removeChannel(channel) }
```

**RLS requirement:** `ALTER TABLE captures REPLICA IDENTITY FULL` in migration 002 (without this, Realtime + RLS silently drops rows).

**WAL trade-off:** `REPLICA IDENTITY FULL` increases WAL size per row. Acceptable at current capture volume. Revisit above ~100K rows/day.

**Scope:** INSERT events only for this plan. UPDATE/DELETE events deferred.

## Forward Compatibility Decision

**`source_account_id` stored now for Phase 3 feature.** The Text pipeline (full post fetch via `source_account_id`) ships in Phase 3. However, the field is cheap to extract now (one SYSTEM_PROMPT line, one nullable column) and eliminates a re-analysis of historical captures later. Deliberate forward-compatibility trade-off: add schema + prompt cost now, avoid historical backfill cost later. If Phase 3 is cancelled, the column can be dropped with a trivial migration.
