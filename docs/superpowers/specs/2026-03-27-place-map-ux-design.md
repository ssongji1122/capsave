# Place Map UX Design — CapSave

## Problem
현재 장소 캡처 카드에 장소당 3개 지도 버튼(네이버/구글/카카오)이 반복되어 UI가 지저분함. 장소 3개면 버튼 9개. 사용자가 "이미 지도에 표시된 상태로 보고 싶다"고 요청.

## Solution Overview
두 가지 뷰를 조합:
1. **캡처 카드 (C 방식)** — 장소를 번호 리스트로 깔끔하게 표시 + "지도에서 보기" 버튼 1개
2. **풀스크린 지도 뷰 (B 방식)** — `/map` 페이지에서 모든 장소를 지도에 핀으로 표시

## 1. 캡처 카드 — 장소 번호 리스트

### Before (현재)
- 각 장소마다 개별 카드 + 3개 지도 버튼 (네이버/구글/카카오)
- 장소 3개 = 버튼 9개 = 시각적 노이즈

### After (개선)
- 장소들을 번호 매긴 리스트로 표시 (1, 2, 3...)
- 각 장소: 번호 원형 배지 + 장소명 + 주소/날짜 메타데이터
- 지도 버튼 3개 × N개 → "🗺 지도에서 보기" 버튼 1개
- 클릭 시 `/map?capture={id}` 로 이동하여 해당 캡처의 장소들만 하이라이트

### 컴포넌트 구조
```
CaptureCard
  ├── Image + Badge (📍 장소 N개)
  ├── Title + Summary
  ├── PlaceList (번호 리스트)
  │   ├── PlaceRow (번호 원형 + 이름 + 메타)
  │   ├── PlaceRow
  │   └── PlaceRow
  ├── MapButton ("지도에서 보기")
  ├── Links (장소 외 일반 링크)
  └── Tags + Date
```

## 2. 풀스크린 지도 뷰 (/map)

### 레이아웃
- **전체 화면 지도** — 페이지 전체를 지도가 차지
- **상단 바** — 지도 API 토글 (네이버 ↔ 구글) + 카테고리 필터 칩
- **바텀 시트** — 하단에서 슬라이드업되는 장소 카드 리스트 (가로 스크롤)
- **핀 팝업** — 핀 클릭 시 정보 팝업 (장소명, 주소, 날짜, 캡처 출처, 길찾기/캡처보기 버튼)

### 지도 API
- **네이버맵** + **구글맵** 토글 전환
- 기본값: 한국 장소 → 네이버맵, 해외 장소 → 구글맵 (자동 감지)
- 사용자가 토글로 수동 전환 가능

### 핀 표시
- 장소 카테고리별 색상 코딩 (place-accent: #34D399)
- 핀에 장소명 라벨 표시
- 같은 캡처의 장소들은 번호 매겨서 연결 표시

### 필터
- 전체 / 카테고리별 (축제, 맛집, 카페 등 — tags 기반)
- 특정 캡처의 장소만 보기 (`?capture={id}` 쿼리)

### 핀 팝업 내용
```
┌─────────────────────┐
│  [썸네일 이미지]      │
│  장소명              │
│  주소 (JetBrains Mono)│
│  날짜                │
│  📸 캡처 제목         │
│  ┌──────┬──────┐    │
│  │길찾기 │캡처보기│    │
│  └──────┴──────┘    │
└─────────────────────┘
```

### 바텀 시트
- 핸들바 (드래그하여 확장/축소)
- 가로 스크롤 장소 카드
- 각 카드: 번호 + 장소명 + 메타데이터
- 카드 탭 시 해당 핀으로 지도 이동 + 팝업 표시

## 3. 네비게이션 연결

### 사이드바
- 기존: Home / Places / Texts
- 추가: **지도** 탭 (`/map`)

### 카드 → 지도
- 캡처 카드의 "지도에서 보기" → `/map?capture={id}`
- 해당 캡처의 장소들만 하이라이트, 나머지는 흐리게

### 지도 → 카드
- 핀 팝업의 "캡처 보기" → 해당 캡처 카드로 이동 (또는 모달)

## 4. 데이터 요구사항

### 현재 PlaceInfo 타입 (변경 불필요)
```typescript
interface PlaceInfo {
  name: string;
  address?: string;
  date?: string;
  links?: string[];
}
```

### 지도 표시를 위해 필요한 추가 데이터
- **위도/경도 좌표** — 현재 없음. 두 가지 접근:
  - A) AI 분석 시 좌표도 추출하도록 프롬프트 수정 (Gemini가 알려진 장소의 대략적 좌표 제공 가능)
  - B) 지도 뷰 진입 시 Geocoding API로 주소/장소명 → 좌표 변환 (네이버/구글 Geocoding)
  - **추천: B** — AI 좌표는 부정확할 수 있고, Geocoding이 더 신뢰할 수 있음

### PlaceInfo 타입 확장 (선택)
```typescript
interface PlaceInfo {
  name: string;
  address?: string;
  date?: string;
  links?: string[];
  lat?: number;  // 위도 (geocoding 후 캐시)
  lng?: number;  // 경도 (geocoding 후 캐시)
}
```

## 5. 기술 스택

- **네이버맵:** `@types/navermaps` + Naver Maps JavaScript API v3
- **구글맵:** `@react-google-maps/api` 또는 Google Maps JavaScript API
- **Geocoding:** 네이버 Geocoding API / Google Geocoding API (서버사이드, API 키 보호)
- **좌표 캐싱:** Supabase에 `lat`, `lng` 컬럼 추가하여 한 번 변환한 좌표 저장
